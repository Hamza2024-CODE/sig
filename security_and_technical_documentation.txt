================================================================================
           منصة SGFEP - التوثيق التقني الأمني ودليل اختبار الاختراق (Pentesting)
================================================================================

تم إعداد هذا المستند كدليل تقني شامل لمهندس الأمن / فاحص الاختراق (Penetration Tester)
لإجراء مراجعة برمجية واختبار أمان بأسلوب الصندوق الأبيض (White-Box Testing) للمنصة.

--------------------------------------------------------------------------------
1. المعمارية الأمنية والضوابط المطبقة (Security Architecture)
--------------------------------------------------------------------------------

[أ] حماية معرّفات الكائنات وصلاحيات الوصول (IDOR Protection)
------------------------------------------------------------
- الثغرة السابقة: كانت المنصة تسمح بالوصول المباشر للملفات والشهادات وكشوف النقاط 
  عبر معرفات رقمية متسلسلة في الرابط (مثل `/print/107`).
- الحل المعماري المطبق حالياً:
  1. تعمية الروابط (URL Obfuscation):
     - تم حظر المعرفات الرقمية الصريحة تماماً في روابط الطباعة. أي طلب يحتوي على معرف 
       رقمي مباشر يتم إرجاع خطأ 404 فوراً.
     - يتم تشفير المعرّف تلقائياً باستخدام الفئة المساعدة `App\Helpers\SecureIdHelper`.
     - خوارزمية التشفير: يتم إجراء عملية XOR على مستوى البايت بين معرّف السجل ومفتاح 
       الملوحة السري المخزن في الخادم (`SGFEP_Secure_Salt_2026`)، ثم يتم تحويل الناتج 
       إلى Base64 آمن للروابط عبر استبدال الرموز (`+` بـ `-` و `/` بـ `_` وحذف `=`).
       مثال: المعرّف `107` يتم تشفيره إلى السلسلة النصية `Yndx`.
  2. التحقق من صلاحيات الكائنات (Object-Level Authorization):
     - بعد فك التشفير بنجاح في المتحكم `ModulesController::printDocument` واستخراج المعرف 
       الحقيقي، يتم تفعيل فحص صلاحيات صارم بناءً على رتبة الجلسة (Session Role):
       - رتبة `etablissement` أو `directeur`: يتم التحقق بطلب SQL من أن معرف المؤسسة 
         المسند للموظف/المتربص يطابق معرف المؤسسة للمستخدم الحالي (`etablissement_id`).
       - رتبة `dfep` (مديرية الولاية): يتم التحقق من أن السجل يتبع لمؤسسة تقع في نفس 
         النطاق الجغرافي للمديرية الحالية (`IDDFEP` يطابق `iddfep`).
       - رتبة `stagiaire` (المتربص): يتم التحقق من أن معرف المتربص يطابق معرف جلسة الطالب نفسه.
       - في حال فشل أي شرط، يتم إرجاع استجابة **`403 Forbidden`** فوراً.

[ب] تشفير البيانات الحساسة عند الخمول (Cryptography at Rest)
-------------------------------------------------------------
- حماية الهوية: يتم تشفير رقم التعريف الوطني (NIN) لجميع الموظفين والمتربصين في قاعدة البيانات.
- التقنية المستخدمة:
  - استخدام نظام التشفير القياسي المتوافق مع Laravel Crypt وهو **AES-256-CBC** مع توقيع 
    رقمي (HMAC) لمنع هجمات التعديل على الشيفرة (Ciphertext Tampering).
  - يتم التشفير وفك التشفير تلقائياً عبر فئة التحويل المخصصة `App\Casts\EncryptedNin`.
  - في استعلامات الطباعة المباشرة (PDO)، يتم استدعاء دالة `Crypt::decryptString` لفك 
    التشفير برمجياً قبل تمرير البيانات إلى محرك التوليد.

[ج] حماية الجلسات وبوابة الدخول (Authentication & Rate Limiting)
-----------------------------------------------------------------
- اختبار الكابتشا (Math CAPTCHA): تفعيل كابتشا رياضية ديناميكية عند تسجيل الدخول لمنع 
  هجمات القوة الغاشمة (Brute-force) والـ Credential Stuffing.
- سجل العمليات (Audit Trail): يتم تسجيل كافة محاولات الدخول الفاشلة والناجحة، وعمليات 
  توليد الوثائق في سجلات الخادم مع تحديد عنوان IP وبصمة المتصفح (User Agent).

--------------------------------------------------------------------------------
2. خريطة مسارات واجهة برمجة التطبيقات (API Routes Map)
--------------------------------------------------------------------------------

جميع المسارات تقع تحت البادئة `/api/v1/` وتتطلب التحقق عبر توكن JWT أو X-API-Key:

1. إصدار التوكن (Token Generation)
   - المسار: `POST /api/v1/auth/token`
   - المعلمات المطلوبة (JSON): `{"api_key": "string"}`
   - آلية العمل: يتم تبديل مفتاح الـ API بتوكن JWT صالح لمدة 8 ساعات.
   - الأمان: يتم مقارنة المفاتيح باستخدام دالة `hash_equals` المقاومة لهجمات التوقيت (Timing Attacks).

2. التحقق من الاتصال (Verify Connection)
   - المسار: `GET /api/v1/verify`
   - الصلاحية المطلوبة: `Authorization: Bearer <token>` أو `X-API-Key: <key>`
   - الاستجابة المتوقعة: تفاصيل الهوية وصلاحيات المفتاح المستخدم.

3. جلب قائمة المتربصين (Trainees List)
   - المسار: `GET /api/v1/stagiaires`
   - الصلاحية المطلوبة: JWT أو API Key
   - المعلمات الاختيارية (Query):
     - `valide` (0 أو 1) - حالة النشاط.
     - `etablissement_id` (int) - فلترة حسب المؤسسة.
     - `specialite_id` (int) - فلترة حسب التخصص.
     - `sexe` (M أو F) - فلترة حسب الجنس.
     - `q` (string) - بحث نصي في الاسم واللقب والـ NCCP.

4. جلب قائمة الموظفين والمؤطرين (Employees List)
   - المسار: `GET /api/v1/employees`
   - الصلاحية المطلوبة: JWT أو API Key
   - المعلمات الاختيارية (Query):
     - `etablissement_id` (int) - فلترة حسب المؤسسة.
     - `grade` (string) - الرتبة البيداغوجية.
     - `fonction` (string) - الوظيفة الحالية.
     - `q` (string) - بحث في الاسم أو اللقب أو البريد أو الـ NIN.

5. حساب الساعات الشاغرة (Vacant Hours)
   - المسار: `GET /api/v1/hr/formateurs/vacant-hours`
   - الصلاحية المطلوبة: JWT أو API Key
   - الاستجابة المتوقعة: مصفوفة بأسماء الأساتذة وعدد الساعات الشاغرة بجدول التوقيت الخاص بهم.

6. تقرير الميزانية المشفر (Budget Report)
   - المسار: `GET /api/v1/finance/reports/budget`
   - الصلاحية المطلوبة: JWT أو API Key
   - الاستجابة المتوقعة: مصفوفة تحتوي على المفتاح `payload` الذي يضم تقرير الميزانية 
     مشفراً بالكامل بخوارزمية AES-256-CBC.

7. تقديم طلب التجهيزات (Equipment Requests)
   - المسار: `POST /api/v1/assets/requests`
   - الصلاحية المطلوبة: JWT أو API Key
   - المعلمات المطلوبة (JSON):
     - `item_name` (string) - اسم الجهاز المطلوب.
     - `quantity` (int) - الكمية المطلوبة.
     - `urgency` (low, medium, high) - درجة الاستعجال.
